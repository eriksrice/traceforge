# Trace Schema

TraceForge v1 records one deterministic LLM/tool incident workflow as append-only JSONL events. The schema is framework-neutral and should map cleanly to Pydantic models in Phase 2.

## Schema Goals

- Make each workflow step replayable from saved inputs, outputs, fixture references, and hashes.
- Preserve prompt, model, tool, and state metadata needed to explain behavior changes.
- Separate protected behavior fields from allowed volatile fields.
- Support first-divergence comparison without depending on LangGraph, live model calls, or external tracing services.

## Event Stream

Each trace file is newline-delimited JSON. Each line is one event. Events are append-only and ordered by `step_index`, then by event creation order.

Expected v1 trace files:

- `traces/baseline_good.jsonl`
- `traces/incident_bad.jsonl`
- `traces/patched_good.jsonl`

These files must be generated by TraceForge commands after implementation. They must not be hand-authored as completed evidence.

## Event Types

Allowed `event_type` values:

- `step_start`
- `model_call`
- `tool_call`
- `state_transition`
- `step_end`
- `replay_comparison`
- `gate_result`

## Step Names

The v1 workflow has exactly three workflow steps:

1. `intake_route`: normalize the support alert and select the first diagnostic tool.
2. `collect_evidence`: call the selected diagnostic tool through the mocked replay layer.
3. `classify_incident`: produce the typed incident packet and response plan.

## Required Core Fields

Every trace event must include:

- `event_id`: stable unique event id.
- `trace_id`: workflow trace id shared by all events in one run.
- `run_id`: concrete execution run id.
- `parent_event_id`: parent event id, or `null` when not applicable.
- `step_index`: integer step order; use `1`, `2`, or `3` for workflow steps.
- `step_name`: one of the v1 step names.
- `event_type`: one of the allowed event types.
- `timestamp_start`: ISO-8601 UTC timestamp.
- `timestamp_end`: ISO-8601 UTC timestamp, or `null` for an open event.
- `latency_ms`: measured duration in milliseconds, or `null` when not applicable.
- `status`: one of `started`, `success`, `error`, `skipped`, or `replayed`.
- `schema_version`: trace schema version, starting with `traceforge.trace_event.v1`.

## Model And Prompt Fields

Model-related events must include:

- `model_provider`: `synthetic` for v1 fixtures; future values may include provider names.
- `model_name`: model or fixture family name.
- `model_version`: model or fixture version.
- `temperature`: numeric temperature, or `null` for synthetic deterministic outputs.
- `seed`: deterministic seed, or `null` when not used.
- `prompt_id`: stable prompt identifier.
- `prompt_version`: prompt version, such as `triage_route_prompt:v1`.
- `prompt_hash`: stable hash of the prompt text or prompt fixture.
- `system_prompt_hash`: stable hash of the system prompt, or `null`.
- `input_hash`: stable hash of the normalized model input.
- `output_hash`: stable hash of the model output fixture.
- `token_input_count`: integer token count, or `null` for synthetic outputs.
- `token_output_count`: integer token count, or `null` for synthetic outputs.
- `estimated_cost_usd`: numeric cost estimate, or `null` for synthetic outputs.
- `model_output_mode`: one of `cached`, `synthetic`, or `live`.
- `model_fixture_id`: fixture id used for v1 replay, or `null`.
- `model_fixture_version`: fixture version used for v1 replay, or `null`.

V1 regression replay must use `cached` or `synthetic` model output mode. `live` is allowed only in non-gating exploratory traces and must be classified as volatile.

## Tool Call Fields

Tool-call events must include:

- `tool_name`: selected tool name.
- `tool_version`: tool contract version.
- `tool_call_id`: stable id for the tool call.
- `tool_input`: structured tool input.
- `tool_input_hash`: stable hash of `tool_input`.
- `tool_output`: structured tool output, or `null` before completion.
- `tool_output_hash`: stable hash of `tool_output`, or `null`.
- `tool_latency_ms`: fixture latency or measured duration.
- `tool_status`: one of `success`, `error`, `timeout`, or `skipped`.
- `tool_error_type`: error label, or `null`.
- `tool_fixture_id`: fixture id used for mocked replay.
- `tool_fixture_version`: fixture version.
- `tool_replay_mode`: one of `mocked`, `live`, or `skipped`.

V1 regression replay must use `tool_replay_mode: mocked`.

## State Transition Fields

State transition and step-end events must include:

- `state_before_hash`: stable hash of state before the transition.
- `state_after_hash`: stable hash of state after the transition.
- `state_patch`: structured patch or summary of changed state fields.
- `state_schema_version`: state schema version, starting with `traceforge.workflow_state.v1`.
- `transition_label`: stable label for the transition.
- `protected_fields`: list of protected field paths considered release-gating evidence.
- `changed_fields`: list of changed field paths.

Required v1 transition labels:

- `route_selected`
- `evidence_collected`
- `incident_classified`

## Replay And Divergence Fields

Replay comparison events must include:

- `baseline_event_id`: compared baseline event id.
- `replay_event_id`: compared candidate event id.
- `replay_mode`: one of `original`, `incident`, or `patched`.
- `replay_status`: one of `matched`, `diverged_allowed`, `diverged_unacceptable`, or `not_replayable`.
- `determinism_class`: one of `deterministic`, `stochastic_stable`, `volatile`, or `unknown`.
- `divergence_label`: one of the labels listed below.
- `divergence_severity`: one of `none`, `info`, `warning`, or `blocking`.
- `first_divergence`: boolean.
- `root_divergence_event_id`: event id for the root divergence, or `null`.
- `allowed_divergence_reason`: explanation for allowed divergence, or `null`.

Allowed `divergence_label` values:

- `none`
- `allowed_nondeterministic`
- `prompt_output_changed`
- `tool_selection_changed`
- `tool_response_changed`
- `state_transition_changed`
- `schema_violation`
- `missing_event`
- `unexpected_error`

The seeded incident must report first unacceptable divergence as `tool_selection_changed` at Step 1 field `requested_tool`.

## Governance Fields

Gate and decision events must include:

- `incident_id`: incident id, starting with `checkout_timeout_wrong_tool`.
- `decision_record_id`: release or review decision id, or `null`.
- `release_candidate_id`: candidate id under evaluation, or `null`.
- `gate_name`: gate name, or `null`.
- `gate_status`: one of `pass`, `fail`, or `warn`, or `null`.
- `review_required`: boolean.
- `notes`: short human-readable notes.

## Protected Fields

Protected fields are release-gating behavior fields. Any change is unacceptable unless explicitly allowed by the replay contract.

V1 protected fields:

- `step_1.output.requested_tool`
- `step_1.output.diagnostic_route`
- `step_1.state.transition_label`
- `step_2.tool_name`
- `step_2.output.evidence_family`
- `step_2.state.transition_label`
- `step_3.output.incident_type`
- `step_3.output.severity`
- `step_3.output.engineering_next_action`
- `step_3.output.escalation_required`
- `step_3.state.transition_label`
- `trace.schema_valid`

## Allowed Volatile Fields

These fields may vary without failing the gate when protected fields match:

- `event_id`
- `timestamp_start`
- `timestamp_end`
- `latency_ms`
- `tool_latency_ms`
- `estimated_cost_usd`
- `token_input_count`
- `token_output_count`
- Nonprotected explanatory prose.
- Report formatting generated from the same trace facts.

## V1 Expected Values

Baseline and patched protected values:

- `requested_tool`: `service_metrics_lookup`
- `diagnostic_route`: `service_health`
- `evidence_family`: `service_telemetry`
- `incident_type`: `service_regression_after_deploy`
- `severity`: `high`
- `escalation_required`: `true`

Seeded bad protected values:

- `requested_tool`: `billing_ledger_lookup`
- `diagnostic_route`: `billing_account_review`
- `evidence_family`: `billing_ledger`
- `incident_type`: `account_billing_configuration_issue`
- `severity`: `medium`
- `escalation_required`: `false`
